# 模块边界的量化评估——让系统性问题对 LLM 可见

> 来源：007（模块边界形态）的延伸。用户指出"需要想想如何规范化，让系统性问题对LLM可见（类似加权评分）"。

## 问题

012 的评分模型工作在**函数粒度**——单个函数的加权行数 × 能力数。但有些代码质量问题不体现在单个函数上，而是体现在**模块之间的关系**上：

- 循环依赖
- 模块间耦合度（A 模块的函数调用 B 模块的函数的次数）
- 边界层是否存在（外部 IO 是否经过 adapter 还是直接散布在业务逻辑中）
- 能力泄漏（一个声明为 pure 的模块里是否间接调用了 IO 函数）

这些是**系统性问题**，单看任何一个函数都看不出来，需要跨模块分析。

LLM 在写代码时通常一次只看一个文件，对跨文件的系统性问题天然盲区。如果评分系统能量化这些问题并报告出来，LLM 就有了"全局视角"。

## 可能的度量方向

### 1. 模块能力轮廓

每个模块（文件或目录）的能力得分分布。一个理想的模块要么是"纯计算模块"（所有能力得分为 0），要么是"IO 边界模块"（只有 I/A 得分，且很薄）。

如果一个模块同时有大量纯计算得分和 IO 得分，说明关注点没有分离。

```
模块能力轮廓：
  src/domain/order.ts     → pure(180) E(20)        ✅ 几乎纯计算
  src/infra/db.ts         → I(50) A(50) E(30)      ✅ IO 边界
  src/handlers/api.ts     → pure(100) I(80) A(80)  ⚠ 混合了计算和IO
```

### 2. 能力边界清晰度

项目中 IO 函数的调用是否都经过了 adapter 层（011 的 `.caps.ts` 声明机制），还是直接 `import { readFileSync } from "node:fs"` 散布各处。

```
边界清晰度：
  直接外部 IO 调用: 12 处   ⚠
  经过 adapter 层: 5 处     ✅
  建议: 将 12 处直接调用迁移到 adapter
```

### 3. 跨模块能力传播

追踪能力在模块间的传播路径。如果 A 模块是 pure 的，但它依赖的 B 模块有 IO，那 A 的"真实"能力集应该包含 IO——这就是能力泄漏。

### 4. 依赖方向一致性

类似于 Clean Architecture 的依赖规则：高层模块不应该依赖低层实现。可以用能力得分来定义"层级"——纯计算是最高层，IO 是最底层。依赖方向应该是从高能力模块指向低能力模块。反向依赖意味着架构问题。

## 与 012 的关系

012 的函数级评分是基础层。模块级评分是在其之上的聚合层：

```
行级: line_weight = 1 + depth + branch_bonus
函数级: weighted_lines × capability_count      ← 012 已实现
模块级: 模块能力轮廓 + 边界清晰度 + 传播分析   ← 本篇讨论
项目级: 模块间依赖方向一致性                    ← 最终目标
```

## 待讨论

1. 模块级评分是否应该产出一个数字（类似函数级总分），还是产出一个结构化报告（轮廓图 + 告警列表）？
2. 实现优先级：是否等 012 的函数级评分稳定后再做，还是可以并行推进？
3. 能力传播分析需要跨文件的 import 追踪——技术上 ESLint 可以做到（它有 module resolver），但复杂度显著高于单文件分析。

## 我的看法

其实这里有一个很巧妙的地方：

- 循环依赖会导致 两方的权限都减不下去，无法拆分
- 模块间耦合度也是一样的，聚类应该能够在我们的评分体系里将这一部分的得分降低。
- 能力泄漏是标准化问题，比较容易从其调用的函数推理出来，这里可以考虑自动化校验（但是当前的校验系统强迫标注往大了标注，因为反向会报错，多的标注会导致加分，少的标注会导致报错）

## AI 回复

三个观察都很精准，逐个展开。

### 循环依赖 → 权限互锁

你说的"两方的权限都减不下去"完全正确。具体机制是：

```
模块 A 有函数 a_IO，调用了模块 B 的 b_IO_Fallible
模块 B 有函数 b_IO_Fallible，调用了模块 A 的 a_IO
```

在能力评分中，A 的 IO 得分包含 a 调用 b 带来的行数，B 的 IO 得分包含 b 调用 a 带来的行数。要降低 A 的 IO 得分，必须把 b 的 IO 隔离出来——但 b 又依赖 a。互锁了。

这意味着**循环依赖会让评分模型的激励梯度失效**——无论怎么拆单个函数，模块级的得分降不下来。这本身就是一个有价值的信号：如果一个模块的得分怎么优化都降不动，很可能是因为有循环依赖。

不需要专门检测循环依赖——评分的"降不动"就是信号。

### 耦合度 → 聚类自然降分

也对。如果 A 模块有 10 个函数都调用 B 模块的 IO 函数，那 A 的 IO 得分会很高。把 A 中和 B 交互的部分聚合成一个 adapter 函数，其余 9 个函数改为调用这个 adapter → IO 被隔离到 1 个函数中 → A 的 IO 得分大幅下降。

评分模型已经在激励这个行为，不需要额外的"耦合度指标"。

### 能力泄漏 → 标注膨胀问题

这是你指出的最关键的点。当前系统的矛盾：

```
规则：调用方能力 ⊇ 被调方能力，否则报错
结果：标注只会越来越大，不会缩小
```

如果一个函数调用了 IO 函数但自己只标了 `@capability Fallible`，lint 报错，逼你加上 `IO`。这确保了正确性。但副作用是——开发者为了不报错会往大了标注（比如直接标 `@capability IO Async Fallible` 全家桶），而"多标"不报错，只是在评分中多扣分。

这其实是评分模型发挥作用的地方：多标导致高分，而评分给出了"你标多了，可以通过拆分来减少"的反馈。**lint 保证下限（不能少标），评分激励上限（不应该多标）**。两者配合才完整。

但这里确实有一个自动化机会：既然能力可以从调用链推断出来，系统可以提供一个"能力推断"工具——扫描函数体中所有调用，计算实际需要的最小能力集，然后和声明的能力集对比。如果声明 > 实际需要，提示"你可以缩减标注"或"你可以拆分函数来减少能力"。

```
⚠ registerUser 声明了 [IO, Async, Fallible, Blocking]
  实际调用链需要: [IO, Async, Fallible]（hashPassword 是 Blocking 但可以拆出去）
  建议: 将 hashPassword 提取为独立函数，registerUser 可以去掉 Blocking
  预期得分: 从 93 降到 72（-23%）
```

这就是"让系统性问题对 LLM 可见"的具体形态——不只是报告问题，还给出具体的优化路径和预期收益。
