# 能力加权评分模型——量化驱动 LLM 优化代码

## 背景

011 建立了能力标注体系和 ESLint 规则，但健康报告只统计"函数个数"——纯函数数占总函数数的比例。把依赖注入提取出纯逻辑后，IO 函数数量不变，好函数比例几乎没有变化。系统缺少正向反馈。

真正有意义的不是"有多少个函数"，而是"多少代码处于什么能力负担之下"。一个 400 行的 ABEMI 巨函数和一个 3 行的 IO 薄包装，不应该在报告里占相同权重。

## 核心思路

将圈复杂度和代码行数统一为一个**逐行加权**指标，然后按能力聚合。

### 公式

每行代码的权重：

```
line_weight = 0                          （空行 / 注释 / 纯花括号）
line_weight = 1 + nesting_depth + 0.5    （分支行：if / for / catch / case 等）
line_weight = 1 + nesting_depth          （普通代码行）
```

`nesting_depth` 由 AST 分析确定——一行代码被多少层 `if / for / while / try / catch / switch` 包围，就有多大的嵌套深度。

函数的加权行数：

```
weighted_lines(fn) = Σ line_weight(i)    对 fn 内的每一行 i
```

单个能力的得分：

```
cap_score[C] = Σ weighted_lines(fn)      对所有携带能力 C 的函数 fn
```

项目总分：

```
total = Σ cap_score[C]                   对所有能力 C
```

### 为什么不用整函数的圈复杂度

"这个函数 CC=25" 是一个定性标签，无法反映内部的不均匀性。同一个函数里，深层嵌套 catch（6 层深）的那行代码和顶层的 `return user` 代价完全不同。

逐行加权自然地捕捉了这个差异：

```
  48 d0   1.0  registrationStats.total++;           // 顶层，权重 1
 163 d6*  7.5  } catch (updateErr) {                // 6层嵌套，权重 7.5
```

同时，逐行加权的聚合结果和整函数圈复杂度高度相关——高 CC 的函数一定有深嵌套，深嵌套一定推高逐行权重。但逐行版本更细腻，不会因为"很多平行 if"（嵌套浅但 CC 高）而过度惩罚。

### 未声明函数的惩罚

未声明能力的函数被视为全能力（携带所有能力），加权行数同时计入每一个能力的得分。这是最严厉的惩罚——一个 400 行的未声明函数对总分的贡献是声明了单一能力的同规模函数的 8 倍。

## 实验验证

用同一个业务场景（用户注册流程）写了三个版本，逐步优化：

### 改造前（example-bad.ts）

一个未声明的巨函数 `registerUser`，145 行有效代码，CC=45，最大嵌套 6 层。输入校验、密码哈希、数据库操作、推荐码处理、邮件发送全部混在一起。

### 改造后（example-good.ts）

按能力边界拆成 8 个函数：`validateUserInput`（E）、`hashPassword_Blocking`（B）、`buildWelcomeEmail`（pure）、4 个 IO 函数（AEI / AI）、1 个协调层（AEI）。

### 最优化（example-best.ts）

在 good 基础上进一步拆细：校验拆 5 个子函数（每个 CC ≤ 7），IO 薄层化（每个只做一件事），提取 4 个纯函数（推荐码逻辑、邮件内容、ID 生成），协调层尽可能薄。共 18 个函数。

## 关键发现

### 1. 能力隔离是主导因素

bad → good（-92%）贡献了绝大部分降幅。这一步做的是"声明能力 + 按能力边界拆分"。评分模型最大幅度地奖励了这一步，与直觉一致：

- 未声明巨函数的惩罚极大（每行代码×8 种能力）
- 拆分后每个函数只携带必要的能力，大量行数从不相关的能力得分中消失

### 2. 极致优化有回报但边际递减

good → best 只降了 19%（266 → 215.5），但做了大量工作（校验拆细、IO 薄层化、纯函数提取）。这不是模型的缺陷——而是正确反映了现实：第一步（声明 + 拆分）收益巨大且容易做到，第二步（极致优化）边际递减。评分模型不会过度激励无意义的拆分。

### 3. 纯函数贡献为零

`buildWelcomeEmail`（good 版本）和 `emailBody`（best 版本）都有 CC=7~8、加权行数 16~29.5，但因为是 pure（无能力），对总分贡献为 0。这精确地反映了：纯函数不增加系统的能力负担。把逻辑从有能力的函数中提取成纯函数，是"免费"的优化——代码总量不变，但得分下降。

### 4. IO 层越薄得分越低

best 中 `queryUserByEmail_IO_Async` 只有 2 行加权，而 `insertUser_IO_Async` 有 9.5。薄 IO 层 = 更好的关注点分离，得分正确反映了这个差异。

### 5. 协调层是不可压缩的成本底线

`registerUser_IO_Async_Fallible` 在 best 中占总分的 43%（31 加权行 × 3 能力 = 93）。它必须组合 Async + IO + Fallible，CC=6 已经很低了。这就是业务的固有复杂度——评分模型承认它的存在而不试图消灭它。

### 6. 嵌套深度的非线性惩罚自然生效

不需要额外的超线性权重函数。通过 `weight = 1 + depth`，一行 depth=6 的代码权重 7.0（是顶层的 7 倍），而拆成多个浅函数后大部分代码 depth=0~1，权重 1~2。这个差异足以产生显著的激励梯度。

## E（Fallible）的特殊性

E 在 best 中仍是最大的单项得分（87，占总分 40%）。来源分两部分：

- **校验函数**（47 分）：5 个 validate 函数天然携带 E，这是校验的本质
- **IO + 协调层**（40 分）：网络调用可能失败，协调层组合了 Fallible 的 IO

E 与其他能力的根本区别：**E 是可逆的**。

```typescript
// get 有 E：可能找不到用户
function getUser_IO_Fallible(id: string): User | null { ... }

// getOrCreate 没有 E：保证返回用户
function getOrCreateUser_IO(id: string): User { ... }
```

E 不是代码本身的固有属性，而是"未被处理的可能性"。一旦在边界层把 E 处理掉（try-catch、类型守卫、Parse don't validate），下游所有函数不需要携带 E。

Parse don't validate 的终极价值不在于消除 validate 函数的 E（它该有），而在于**阻止 E 向下游传播**。best 版本已经做到了这一点：下游的纯函数和 IO 薄层都不携带 E。

### 能力的可消除性分类

| 分类 | 能力 | 消除方式 |
|------|------|---------|
| 可包装消除 | E（Fallible） | Parse don't validate、getOrCreate、边界 try-catch |
| 可重写消除 | M（Mutable） | 返回新对象替代修改参数 |
| 可重写消除 | P（Impure） | 依赖注入替换全局状态 |
| 只能隔离 | A（Async）、B（Blocking）、I（IO） | 提取到独立函数，缩小携带面积 |
| 固有标记 | T（ThreadLocal）、U（Unsafe） | 无法消除，只能标记和隔离 |

E 的"可包装消除"和 M 的"可重写消除"有本质区别：

- E 可以通过**包装**消除——原函数不改，加一层边界就行
- M 必须**重写**原函数本身（改为返回新对象而不是修改参数）

这个分类对 LLM 提供优化建议时直接可用：看到 E 得分高，建议 Parse don't validate；看到 M 得分高，建议纯函数化；看到 I 得分高，建议 IO 薄层化。

## 实现

评分引擎在 `prototype/scoring/engine.ts`，基于 `@typescript-eslint/parser` 做 AST 分析：

1. 解析源码得到完整 AST
2. 递归遍历 AST，对每一行标记嵌套深度和分支标记
3. 提取函数边界和能力声明（后缀 / JSDoc）
4. 按函数聚合加权行数，再按能力聚合得分

对比脚本在 `prototype/scoring/compare.ts`，三个示例文件在同目录下。

## 待探索

1. **E 的传播得分**：当前模型中"已被边界吸收的 E"和"向下游泄漏的 E"得分相同。是否需要区分——泄漏的 E 应该更贵？
2. **能力分类的正式化**：将可消除性分类写入能力词汇表（`capabilities.ts`），影响报告输出和优化建议。
3. **集成到 ESLint rule**：当前是独立脚本，后续应作为 ESLint processor 或 rule 运行，与 no-escalation 规则共存于 CI 流水线。
4. **真实项目验证**：在一个非玩具项目上运行评分，观察得分分布是否符合直觉。
