# 能力分类：重新定义能力词汇表

> 来源：012 提出了可消除性分类，用户在早期讨论中指出 E 的可逆性和 M 的特殊性。本篇集中讨论能力应该如何分类。

## 当前状态

### 现有 8 个能力

```
Async       异步函数，包含 await
Blocking    可能阻塞当前线程
Fallible    可能返回错误 / 抛出异常
IO          执行 I/O 操作（网络、文件、数据库）
Mutable     修改参数中的可变状态
Impure      有副作用（全局变量、环境变量、随机数等）
ThreadLocal 依赖线程局部状态
Unsafe      包含不安全操作
```

### 好函数判定

能力 ⊆ {Async, Blocking, Fallible, Mutable} 的函数为"好函数"（方便单测）。

## 存在的问题

### 1. 能力之间有强共现关系

从我们三个示例文件的实际标注统计：

```
IO 出现 19 次，其中：
  12 次和 Async 共现（63%）→ JS 中 IO 几乎就是 async 的
   8 次和 Fallible 共现 → IO 操作通常可能失败

Async 出现 12 次，其中：
  12 次和 IO 共现（100%）→ 在我们的示例中，Async 从未独立于 IO 出现
```

这意味着标注 `_IO_Async_Fallible` 时，`Async` 和 `Fallible` 大部分是冗余的——它们几乎总是 IO 的附属品。

### 2. 在 JS/TS 中部分能力没有意义

- **ThreadLocal**：JS 是单线程的，Worker 线程是隔离的。在 TS 生态中几乎不出现。
- **Unsafe**：JS 没有 `unsafe` 关键字。`eval` 和 `as any` 可以归入其他类别（Impure 和类型松散度）。

### 3. 边界模糊

- **Impure vs IO**：`Date.now()` 算 Impure 还是 IO？`Math.random()` 呢？`console.log()` 呢？Impure 被定义为"IO 之外的副作用"，但边界不清晰。
- **Async vs Blocking**：它们是同一个操作的两种执行方式（`readFile` vs `readFileSync`），而非两种独立能力。

### 4. 可消除性没有反映在分类中

012 提出了可消除性维度，但它目前只是一个分析结论，没有融入能力词汇表本身。

## 从第一原理重新审视

### 能力的本质是什么？

能力标注回答的问题是：**调用这个函数时，调用者需要承担什么额外负担？**

- 调用一个纯函数：零负担（给输入，得输出，无副作用）
- 调用一个 IO 函数：需要考虑网络是否可达、数据库是否连接
- 调用一个 Fallible 函数：需要处理错误路径
- 调用一个 Mutable 函数：需要追踪被修改的状态

### 评分模型关心什么？

从 017 的统一评分模型看，能力影响得分的方式是：`weighted_lines × capability_count`。能力越多，同样的代码行数贡献的得分越高。

所以能力分类的设计目标是：
1. **每个能力应该代表一种独立的"负担"**——不应该有强共现（否则等于重复计分）
2. **LLM 应该能针对每个能力采取独立的优化行动**——否则这个能力没有指导价值
3. **在 JS/TS 生态中有实际意义**——去掉在目标语言中不存在的能力

### 重新推导

按"调用者的独立负担"分类：

**负担 1：需要处理外部世界的不确定性**
→ IO（读写外部系统）
在 JS 中 IO 通常意味着 async，且通常可能失败。但 Async 和 Fallible 不是独立负担——它们是 IO 的附属特征。

**负担 2：需要处理错误路径**
→ Fallible（可能失败）
但这里有细分：
- IO 导致的 Fallible（网络超时）——这是 IO 的附属，不是独立负担
- 业务逻辑导致的 Fallible（输入校验失败）——这是独立负担，可以通过 Parse don't validate 消除

**负担 3：需要追踪被修改的状态**
→ Mutable（修改参数/外部状态）

**负担 4：需要管理执行模型**
→ Async（需要 await，影响调用者的控制流）
→ Blocking（阻塞线程，影响调用者的性能）
这两个确实是独立负担——调用者需要决定用 await 还是放到 worker 中。

**负担 5：需要注意隐式依赖**
→ Impure（依赖全局状态、时间、随机数）
和 IO 的区别：IO 是显式地和外部系统交互，Impure 是隐式地依赖环境。

## 提案：精简到 5 个核心能力

去掉 ThreadLocal（JS 不需要）和 Unsafe（JS 中定义模糊，可归入类型松散度维度），剩余 6 个合并/保留为 5 个：

| 能力 | 含义 | 消除方式 | 优化建议 |
|------|------|---------|---------|
| **IO** | 读写外部系统（网络、文件、数据库） | 只能隔离 | IO 薄层化、adapter |
| **Fallible** | 可能失败（校验失败、解析失败） | 可包装消除 | Parse don't validate |
| **Mutable** | 修改参数或外部可变状态 | 可重写消除 | 返回新对象 |
| **Async** | 需要 await，影响调用者控制流 | 只能隔离 | 提取异步部分 |
| **Impure** | 依赖隐式环境（时间、随机、全局变量） | 可重写消除 | 依赖注入 |

### 去掉了什么

- **Blocking**：在 JS 中，Blocking 几乎只出现在 `xxxSync` 函数和 CPU 密集计算中。前者应该直接用 async 版本替代（这是优化而非标注），后者可以用 Async（放到 worker）处理。不需要独立能力。
- **ThreadLocal**：JS 单线程，不适用。
- **Unsafe**：JS 没有 unsafe 概念。`eval` 归入 Impure，`as any` 归入类型松散度评分。

### Async 和 IO 的关系

不再假设 IO 隐含 Async。理由：

- 存在同步 IO（`readFileSync`），标注为 `IO`，不带 `Async`
- 存在不是 IO 的 Async（`setTimeout` 包装、`Promise.all` 编排），标注为 `Async`，不带 `IO`
- 大部分 IO 确实是 Async 的，这时标注 `IO_Async`——共现是事实描述，不是冗余

### Fallible 和 IO 的关系

IO 函数通常可能失败，但不总是。比如 `console.log` 是 IO（写 stdout）但几乎不会失败。所以 Fallible 不应该被 IO 隐含——需要时显式标注 `IO_Fallible`。

但 Fallible 的特殊性仍然在于它的可消除性：IO 导致的 Fallible 可以在边界层被 try-catch 吸收，校验导致的 Fallible 可以被 Parse don't validate 消除。

## 消除性分类（融入词汇表）

| 消除性 | 能力 | 机制 | 评分影响 |
|--------|------|------|---------|
| **可包装消除** | Fallible | 边界吸收、Parse don't validate | 下游函数去掉 E → 得分下降 |
| **可重写消除** | Mutable、Impure | 纯函数化、依赖注入 | 函数本身去掉 M/P → 得分下降 |
| **只能隔离** | IO、Async | 薄层化、提取到独立函数 | 携带面积缩小 → 得分下降 |

消除性告诉 LLM **怎么降分**：

- 看到 Fallible 得分高 → 先检查是否有 E 在不必要地传播 → Parse don't validate
- 看到 Mutable 得分高 → 重写为返回新对象的纯函数
- 看到 IO 得分高 → 把 IO 集中到尽可能少的薄函数中
- 看到 Impure 得分高 → 把 Date.now()、Math.random() 改为依赖注入
- 看到 Async 得分高 → 把异步逻辑提取到独立函数，减少 async 的"传染面积"

## 对评分模型的影响

### 017 的能力维度

从 8 个能力变为 5 个。对评分公式无影响——仍然是 `Σ weighted_lines × capability_count`。但：

- 未声明函数的惩罚从 ×8 变为 ×5（全能力种类减少）
- 共现标注变少（不需要重复标 IO_Async_Fallible 中 IO 隐含的部分）

### "好函数"判定是否仍需要？

在评分模型中，"好函数"（能力 ⊆ 某个集合）的概念可能不再需要：

- 旧模式：好函数/坏函数的二元判定 → 和 lint 一样的门禁思维
- 新模式：能力得分是连续的 → 不需要划线，得分低就是好

如果仍然需要一个直觉性的标签，可以说：**纯函数（零能力）= 最好，每增加一个能力就增加一份负担。**

## 待讨论

1. **Blocking 是否真的可以去掉？** 如果项目中有大量 CPU 密集计算（加密、压缩），Blocking 作为独立能力有标记价值。是否要保留为可选能力？

2. **5 个能力是否太少？** 从 8 个减到 5 个，是否丢失了有用的区分？比如去掉 Unsafe 后，`eval` 只被标记为 Impure——这是否足够引起注意？

3. **消除性是否应该影响评分权重？** 可包装消除的 Fallible 是否应该比只能隔离的 IO 权重更低（因为更容易消除）？还是保持统一权重，让 LLM 自己发现哪个更容易降分？

4. **能力词汇表是否应该按语言生态定制？** 当前是为 JS/TS 设计的。Rust 项目可能需要 Unsafe 和 ThreadLocal，Java 项目可能需要 Blocking。是否提供一个可配置的能力集？

## 我的看法

我们这里做的是tslint，如果在ts上表现的好，我们再去考虑其他的。

现在就是这个能力需要在考虑了，话说有没有相关的论文或者研究系统性讨论过这方面的东西？我们可以作为参考。

因为实际上是梯度下降，两个都做才是得分最低，因此统一参数是减少超参数的体现，能够简化实现。

## AI 回复

### 相关研究

找到了直接对应的学术工作。最相关的是 **Koka 语言**（Daan Leijen，Microsoft Research，2014-至今），它在类型系统层面实现了效果推断。Koka 的效果分类和我们的 5 能力提案几乎一一对应：

| Koka 效果 | 我们的能力 | 含义 |
|-----------|-----------|------|
| `io` | IO | 读写外部系统 |
| `exn` | Fallible | 可能抛出异常 |
| `st` | Mutable | 读写可变状态 |
| `async` | Async | 异步操作 |
| `ndet` | Impure | 非确定性（随机、时间等） |
| `div` | （无） | 可能发散（死循环）——TS 中不常见 |

Koka 的关键洞察和我们一致：**效果可以被 handler 消除**。在 Koka 中 `with handler` 可以拦截效果，在我们的系统中 Parse don't validate / try-catch 边界可以吸收 Fallible。这是"可包装消除"的学术对应。

更广泛的背景：

- **代数效果**（Plotkin & Power 2003，Bauer & Pretnar 2015）：效果是一等公民，函数签名声明可能产生哪些效果，handler 可以在外层消除效果。我们的能力标注是代数效果的轻量实现——不需要语言支持，用命名 + lint 模拟。
- **Nielson & Nielson (1999) 效果系统综述**：经典效果分类为 读/写存储、异常、IO、非确定性、发散——和我们的 5 能力几乎一致。
- **Java Checked Exceptions (1996)**：只标注了异常一种效果，因为对人太繁琐被抛弃——正好印证我们的总命题"LLM 时代可以复活对人繁琐的标注"。

### 5 个能力已是合理的最小集

对"能合并就合并"做了验证：

- IO + Async → 不宜合并。`readFileSync` 是 IO 不是 Async；`setTimeout` 包装是 Async 不是 IO。存在不共现的情况。
- Mutable + Impure → 不宜合并。优化手段不同（M 返回新对象，P 依赖注入），合并会丢失"该怎么降分"的指导信息。

5 个能力和 Koka 的 5 个核心效果（去掉 `div`）完全对齐。有学术系统背书，且在我们的实验中每个能力都有独立的优化手段。进一步合并会降低指导价值。

### 聚焦 TS

确认：ThreadLocal、Unsafe、Blocking 在 TS 生态中没有独立存在的意义，去掉。如果未来扩展到 Rust/Java，可以从 Koka 的效果体系中按需加回。

### 统一权重 = 减少超参数

当前 5 个能力在评分中权重相同（每个能力贡献 `weighted_lines` 分）。不需要为不同能力设置不同权重——权重差异会引入需要校准的超参数。保持统一：一个能力 = 一份负担。LLM 在优化时会自然发现哪些能力更容易消除（Fallible 通过 Parse don't validate），哪些只能隔离（IO），不需要我们用权重告诉它。
