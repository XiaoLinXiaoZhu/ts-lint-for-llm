# 代码走查 / 桌面检查 (Desk Checking / Code Walkthrough)

## 是什么

在不运行程序的情况下，开发者（或一组人）手工追踪代码的执行过程：选取一组输入，在纸上或白板上逐行模拟变量变化，验证程序是否产生预期输出。本质上是"人肉 debugger"。

分两种形式：
- **个人桌面检查 (Desk Checking)**：一个人对着自己的代码逐行追踪
- **团队走查 (Walkthrough)**：作者向团队讲解代码执行过程，团队成员提问和发现问题

## 历史相关渊源

在计算机稀缺且昂贵的 1960–1970 年代，程序员无法随时运行代码。上机时间是按小时分配的。为了不浪费宝贵的上机时间，程序员必须在纸上验证程序的正确性——桌面检查是必修技能。

1968 年 NATO 软件工程会议（标志着"软件危机"概念的诞生）上，多位演讲者强调了代码走查作为质量保证手段的价值。

Gerald Weinberg 的 *The Psychology of Computer Programming* (1971) 深入分析了代码走查的心理学：让作者向他人解释代码的过程本身就能发现 bug（后来被命名为"小黄鸭调试法"的原型）。

1980 年代后，随着个人电脑普及和 IDE 调试器的成熟，桌面检查逐渐消失。"直接运行看结果"取代了"在纸上推理"。

## TypeScript 代码举例

```typescript
// ---- 桌面检查过程示例 ----

// 被检查的代码：合并两个有序数组
function mergeSorted(a: number[], b: number[]): number[] {
  const result: number[] = [];
  let i: number = 0;
  let j: number = 0;

  while (i < a.length && j < b.length) {
    if (a[i] <= b[j]) {
      result.push(a[i]);
      i++;
    } else {
      result.push(b[j]);
      j++;
    }
  }

  while (i < a.length) { result.push(a[i]); i++; }
  while (j < b.length) { result.push(b[j]); j++; }

  return result;
}

// ---- 桌面检查追踪表 ----
// 输入: a = [1, 3, 5], b = [2, 4]
//
// | Step | i | j | a[i] | b[j] | Condition    | Action           | result         |
// |------|---|---|------|------|--------------|------------------|----------------|
// | 0    | 0 | 0 | 1    | 2    | 1 <= 2 → T  | push(1), i++     | [1]            |
// | 1    | 1 | 0 | 3    | 2    | 3 <= 2 → F  | push(2), j++     | [1,2]          |
// | 2    | 1 | 1 | 3    | 4    | 3 <= 4 → T  | push(3), i++     | [1,2,3]        |
// | 3    | 2 | 1 | 5    | 4    | 5 <= 4 → F  | push(4), j++     | [1,2,3,4]      |
// | 4    | 2 | 2 | 5    | -    | j>=b.len → ⊥| exit main loop   | [1,2,3,4]      |
// | 5    | 2 | - | 5    | -    | i<a.len → T | push(5), i++     | [1,2,3,4,5]    |
// | 6    | 3 | - | -    | -    | i>=a.len → ⊥| exit cleanup     | [1,2,3,4,5]    |
//
// 最终结果: [1,2,3,4,5] ✓ 正确

// ---- 通过桌面检查发现的 bug 示例 ----
// 如果原代码写成 a[i] < b[j] (没有 =)，桌面检查 a=[1,1], b=[1] 会发现：
// Step 0: a[0]=1, b[0]=1, 1<1=F → push(b[0]=1), j++ → result=[1]
// Step 1: a[0]=1, j>=b.len → exit → cleanup push a[0]=1, a[1]=1 → result=[1,1,1]
// 结果正确！但稳定性被破坏了：相等元素从 b 先入而非从 a 先入。
// 这个 bug 只通过桌面检查的细致追踪才能发现。
```

## 为什么被抛弃

1. **调试器更快**：打断点、单步执行、观察变量——比在纸上追踪快得多
2. **规模不可扩展**：手工追踪 20 行代码可以，200 行就不现实了
3. **痛苦且无聊**：逐行追踪变量变化是极其枯燥的工作
4. **单元测试替代**：写一个测试用例来验证，比手工追踪更可靠且可重复
5. **开发节奏不允许**：现代开发的"写-运行-修"循环以秒计，没有人愿意花 30 分钟做桌面检查

## 是否应该在 LLM 时代重新启用

**核心思想应该复兴——LLM 是完美的"桌面检查执行者"。** 理由：

- LLM 可以瞬间对一段代码做桌面检查：给定输入，逐行追踪变量变化，生成追踪表。这比运行代码更有价值，因为追踪表展示了**为什么**结果是对的（或错的），而不仅仅是结果。
- LLM 做桌面检查时可以自动选取边界输入（空数组、单元素、重复元素、最大值），系统地覆盖极端情况——人类做桌面检查时通常只会测试"正常"输入。
- 对 LLM 自己生成的代码做桌面检查是一种**自我验证机制**：生成代码 → 用不同的推理路径追踪执行 → 检查是否一致。两次独立推理得到相同结果比一次推理更可靠。
- 桌面检查生成的追踪表本身就是极好的文档——新人读追踪表比读代码更容易理解算法。

建议：对算法代码，让 LLM 在生成后自动做一轮桌面检查（选取 3-5 个代表性输入，生成追踪表）。追踪表作为注释或测试用例的注释保留。
