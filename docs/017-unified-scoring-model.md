# 统一评分模型：用连续梯度驱动 LLM 优化代码

> 来源：012（能力加权评分）+ 013（禁用 bool/null）+ 014（类型松散度）的综合。
> 关键决策：所有代码质量信号都做成**连续评分**而非二元 lint，LLM 通过梯度下降自然找到最优解。

## 动机

### 为什么不做 lint（是/否判断）

lint 是门禁思维：设一个阈值，通过就通过，不通过就报错。问题在于：

- **阈值之争**：多少个可选字段算"太多"？2 个？3 个？5 个？没有客观答案
- **误报即噪声**：lint 报了但不该改 → 开发者/LLM 学会忽略它 → 整个规则失去信任
- **没有梯度**：刚好在阈值以下的代码和远低于阈值的代码，得到同样的"通过"——系统不再激励进一步优化

### 评分即梯度

给出连续数字，不做判断。每个 `any` 贡献 10 分松散度，每一层嵌套让那一行的能力负担权重 +1。LLM 的目标是"两个维度的总分都尽可能低"。

- 不需要阈值——没有"通过/不通过"，只有"更高/更低"
- 不存在误报——`any` 确实比 `User` 松散，可选字段确实比必填字段约束弱，这是事实而非判断
- 每一步优化都有回报——把一个 `any` 换成具体类型，立刻降 10 分

### "是否需要保留"不再是问题

评分模型消解了"这个 bool 参数该不该改成枚举"的争论：

- 如果改成枚举后得分降低了 → 这是正确的优化方向
- 如果改成枚举后得分没变（比如谓词返回值的 bool，不影响任何维度）→ 不需要改
- LLM 自己会发现哪些改动降分、哪些不降分，自动选择有收益的方向

## 两个独立维度

### 维度 A：能力负担（来自 012）

度量：函数携带的运行时行为复杂度。

```
line_weight = 1 + nesting_depth + (is_branch ? 0.5 : 0)
weighted_lines(fn) = Σ line_weight(i)
cap_score[C] = Σ weighted_lines(fn)  对所有携带能力 C 的函数
total_capability = Σ cap_score[C]
```

LLM 降分手段：拆分函数、隔离能力到薄层、提取纯函数、Parse don't validate 消除 E。

### 维度 B：类型松散度（来自 013 + 014）

度量：类型系统的约束缺失程度。

```
looseness = Σ penalty(signal)
```

| 信号 | 权重 | 理由 |
|------|------|------|
| `any` | 10 | 完全无约束 |
| `Record<string, any>` | 8 | 键和值都无约束 |
| `unknown` | 3 | 比 any 好，但仍需运行时收窄 |
| 函数参数 `boolean` | 2 | 调用处信息丢失（013 结论） |
| 每个可选字段 `?` | 1 | 缺失语义可能模糊 |

LLM 降分手段：收窄类型（`any` → 具体类型）、消除不必要的 `?`、bool 参数改枚举、拆分大接口。

### 为什么是两个维度而不合并

它们度量的东西不同，优化手段也不同：

- 能力负担高 → 需要**拆函数**（结构调整）
- 松散度高 → 需要**改类型签名**（接口调整）

合并成一个数字会丢失方向信息——LLM 不知道该拆函数还是改类型。分开报告，LLM 自然对准更高的那个维度优化。

## 实验验证

用同一个业务逻辑（用户创建 + 列表查询）写了四个版本，形成二维对照：

|  | 类型松散 | 类型紧凑 |
|--|---------|---------|
| **能力混合** | bad | mixed-B |
| **能力分离** | mixed-A | good |

### 四版代码说明

**bad**（能力混合 + 类型松散）：一个未声明的巨函数，参数全是 `any`，用 `boolean` 控制行为分支，返回 `any`。

**mixed-A**（能力分离 + 类型松散）：按能力拆成 10 个函数（纯函数、IO 薄层、协调层），能力声明完整。但所有类型仍然是 `any`——拆了结构，没改接口。

**mixed-B**（类型紧凑 + 能力混合）：定义了 `Role` 枚举、`CreateUserInput` 精确接口、`CreateResult` 可辨识联合。但全部逻辑仍在一个未声明的巨函数里。

**good**（能力分离 + 类型紧凑）：两个维度都优化。

### 评分结果

```
              bad    mixed-A  mixed-B    good
──────────────────────────────────────────────
能力负担       756      115      620      115
松散度          65      322       12       24
```

### 关键发现

**1. 两个维度确实独立，不可互相替代。**

- bad → mixed-A：能力从 756 降到 115（-85%），但松散度从 65 **升到** 322（拆成更多函数后 any 出现更多次）
- bad → mixed-B：松散度从 65 降到 12（-82%），但能力仍然 620（没拆函数）
- 只有 good 同时在两个维度都低

**2. 拆函数但不改类型会暴露更多类型债务。**

mixed-A 的松散度（322）比 bad（65）还高。原因：bad 只有 1 个函数，5 次 any；mixed-A 拆成 10 个函数，each 函数的参数和返回值都是 any，出现 32 次。这不是评分的 bug——而是正确的信号：你拆开了函数，但类型债务暴露得更多了，下一步应该收窄类型。

**3. 能力负担是结构维度的主导者。**

mixed-A 和 good 的能力负担完全相同（115），因为函数结构一样。类型改善（any → 具体类型）对能力负担没有影响——两个维度确实正交。

**4. good 是唯一在两个维度都低的版本。**

在二维空间中，good 位于左下角（能力 115，松散 24），是最优位置。LLM 的梯度下降路径：bad → 先拆函数降能力 → 再收窄类型降松散 → 到达 good。

## 维度扩展性

当前两个维度已经覆盖了函数结构和类型质量。未来可能加入第三个维度：

- **模块边界清晰度**（015）：已确认可以从能力负担的模块级聚合中自然浮现，暂不需要独立维度
- **命名/文档质量**：难以自动评分，暂不纳入

两个维度已经足以驱动大部分优化行为。保持精简，不添加无法可靠度量的维度。

## 实现

统一评分引擎在 `prototype/scoring/unified-engine.ts`，四个对比样本在同目录下（`unified-bad/good/mixed-a/mixed-b.ts`），对比脚本 `unified-compare.ts`。

引擎对每个文件输出两个数字：`能力负担总分` 和 `松散度总分`。两个数字独立计算、独立报告。

## 已知局限

### 局部最优陷阱

评分系统是贪心搜索的验证工具：每一步修改都要求分数下降，否则撤回。但系统性重构（打破循环依赖、重新划分模块边界）往往需要"先爬坡再下降"——中间步骤可能暂时增加代码量或引入新接口，导致分数上升。

LLM 自然倾向于做 patch 式优化（声明能力、提取纯函数、收窄类型），这些能通过贪心搜索找到。但真正高收益的重构（消除能力弥散的根因、切断循环依赖）需要架构层面的规划，不是逐步 patch 能达到的。

具体表现：

- 分数降到某个值后不再下降，LLM 反复做无效 patch（如提取模板字符串为纯函数，分数不变）
- 循环依赖导致两个模块的能力互锁，无论怎么拆单个函数都降不动
- 正确的重构路径需要先引入抽象（代码量增加、分数上升），再移除旧代码（分数下降），但 LLM 不会走分数上升的那一步

当前状态：接受这是 MVP 的已知局限。评分系统做好验证，架构规划由人来做。后续可能的补充方向：

1. 诊断报告（能力传播链分析、循环依赖检测）——告诉 LLM 为什么卡住
2. 工作流分层（人做规划、LLM 做执行+验证）
3. 指导 LLM 识别"卡住"状态并切换到规划模式

### 权重待校准

松散度的权重（any=10, unknown=3, ?=1 等）是初始拍定的，需要在真实项目上校准。能力负担的行权重（`1 + depth + branch_bonus`）同理。
