# LLM 时代的软件工程：一场辩论与头脑风暴的总结

> 核心命题：很多过去因"对人类太麻烦"而被放弃的软件工程实践，在 LLM 写代码的时代是否应该重新启用？

本文综合了一场"辩论赛"和多个独立专家的深度分析，试图给出一个既不盲目乐观也不保守畏缩的答案。

---

## 一、共识：哪些实践确实值得复活

以下实践经过正反双方和多角度分析后，被认为是"LLM 时代确实可行且有价值"的：

### 1.1 契约式设计（Design by Contract）

每个函数标注完整的前置条件、后置条件和不变量。Bertrand Meyer 1986 年提出的正确做法，因维护成本太高而从未普及。LLM 可以从实现自动逆推契约，并在每次代码变更时同步更新。

**关键提醒**：契约应由 LLM 生成 + 编译器/SMT 求解器验证，不能仅靠人工审查。

### 1.2 属性测试（Property-Based Testing）

不写具体测试用例，而是写数学性质（如"排序是幂等的"、"排序保持长度"），由框架随机生成海量输入。QuickCheck/Hypothesis 风格。LLM 擅长从函数语义推断可测试的性质。

### 1.3 穷举式错误处理

每个可能失败的函数定义完整的错误类型枚举（而非通用 `Exception`），调用者必须处理每种错误。Rust 的 `Result<T, E>` 模式、或者 Python 的 `Result` 类型。LLM 可以穷举失败模式并生成对应错误类型。

### 1.4 "使非法状态不可表示"的类型设计

用类型系统编码业务规则，使无效状态根本无法通过编译。TypeScript 的可辨识联合（Discriminated Unions）、Rust 的类型状态模式。LLM 可以自动生成这类"类型体操"代码，人类只需看接口。

### 1.5 结构化 API 文档和变更日志

OpenAPI 规格、自动生成的 commit message、changelog。这些"无侵入"实践不改变代码结构，即使失败退出成本也为零。

### 1.6 更强的 lint 和静态分析

启用以前因"误报太多"而被关闭的规则，由 LLM 过滤噪音，只将真正的问题呈现给人类。

### 1.7 数据血缘追踪和自动化数据字典

追踪每个字段从哪来、到哪去、经过什么变换。这是数据层面最被低估的工程实践。大多数生产事故不是代码 bug 而是数据问题。LLM 的"不知疲倦"恰好适合这种追踪+记录任务。

theory

### 1.8 端到端请求追踪

从用户请求入口到数据库写入，每个处理步骤都有追踪 ID 和结构化日志。以前因为太繁琐而没人做到完整。

## 二、辩论焦点：有争议的实践

### 2.1 能力标记 / 效应系统（Capability Markers / Effect Systems）

感觉需要写一个简单的 linter，用来配合这个使用。

优

**正方观点**：用函数名前缀（如 `rvs_get_data_AP`）或注解编码函数的元信息——是否异步、是否有副作用、是否线程安全。能力之间有偏序关系，可静态检查调用链合规性。

**反方质疑**：
- 命名约定≠编译器强制，没有强制力的大规模约定必然腐化。匈牙利命名法在 Windows 代码库中最终被放弃正是因为此。
- 如果标记与实际行为不符（比如把有副作用的标记为 pure），命名比没有还危险。
- LLM 修改代码时比人类更不注意维持命名一致性。

**折中方案**：如果要做效应系统，要么让编译器/类型系统强制执行（如 Rust 的 ownership、Haskell 的 monad、Koka 的 effect types），要么不做。靠命名约定的半吊子方案比没有方案更危险。

### 2.2 极端函数分解

差

**正方观点**：每个函数 3-5 行，只做一件事。LLM 不在乎跳转深度。

**反方质疑**：函数分解需要理解"一件事"的语义边界，LLM 不理解语义。粒度不一致的分解比不分解更混乱。更严重的是，20 层调用栈会让代码变成只有 LLM 能维护的东西。

**折中方案**：保持合理粒度（每个函数 < 30 行），但不追求极端。确保人类 10 分钟能理解任何模块的核心逻辑。

### 2.3 文学编程（Literate Programming）

优？我觉得挺好玩的，留下

**正方观点**：Knuth 1984 年提出的理念，代码和文档交织如学术论文。LLM 天然擅长生成叙述性文字。

**反方质疑**：LLM 生成的叙述是"听起来合理的模式匹配"，不是真正的人类思考逻辑。人类看到"很详细的文档"时会降低警惕。如果文档有微妙的不准确，比没有文档更糟。

### 2.4 自动化架构决策记录（ADR）

差。不要

**正方观点**：每次架构决策自动记录背景、权衡、备选方案。

**反方质疑**：LLM 不知道真实的决策过程（"选 PostgreSQL 是因为团队没人会 MongoDB"——这种信息不在代码里）。LLM 生成的 ADR 可能是"听起来合理但完全编造的理由"。编造历史比没有历史更危险。

### 3.3 推荐工作流：Formality-in-the-Loop

但是其实现在没有功能这么强的检查器。所以还是不行，得靠 llm 来。

```
自然语言需求 → [LLM] → TLA+ 规约 + Z Notation 数据模型
                         ↓
                   [模型检查器验证]
                         ↓
              带契约的代码 → [LLM] → Dafny/Verus 验证关键算法
                         ↓
              自动生成的测试 + 精化类型 → [CI/CD] 持续验证
```

## 四、架构层面的变革

来自资深架构师的分析：

### 4.1 Contract-First Modular Architecture (CFMA)

**核心思想**：合约层由人类编写（每个文件 < 100 行），实现层由 LLM 生成。

```
contracts/          ← 人类写（接口、领域模型、事件定义）
  domain/           ← 核心业务类型
  services/         ← Protocol/Interface 定义
  repositories/     ← 数据访问接口
  events/           ← 事件定义

implementations/    ← LLM 生成（只要通过合约测试即可）
  services/
  repositories/
  gateways/

wiring/             ← 依赖注入配置
tests/
  contracts/        ← 合规性测试（人类写）
  behaviors/        ← 行为测试（人类写）
  implementations/  ← 实现测试（LLM 可生成）
```

**依赖方向**：implementations → contracts（单向）。LLM 生成实现时只需读合约文件。

## 五、被忽视的维度

### 5.1 数据层面（最被低估）

- **数据血缘追踪**：追踪字段的来源、去向和变换。大多数生产事故是数据问题而非代码问题。

## 六、风险与局限（反方核心论点）

### 6.1 上下文窗口是硬约束

元信息（断言、注解、标记、规约）占据代码的 60-70% 时，LLM 的有效"逻辑带宽"反而下降。一个充满复杂注解的代码库，其有效信息密度更低，LLM 能处理的实际复杂度上限反而更低。

### 6.2 复杂度守恒定律

复杂度不会消失，只会转移。从"写代码的人"转移到了"读代码、debug、review、维护系统的人"。如果这些人也是 LLM，回到 6.1。如果是人类，复杂度成本只是被推迟和放大了。

### 6.3 LLM 不真正理解形式化方法

LLM 可以生成看起来正确的 TLA+/Coq 代码，但不真正理解时序逻辑的语义。**错误的形式化规约比没有规约更糟糕**，因为它给了你虚假的信心。

### 6.4 历史教训：EJB 2.x

Enterprise Java Bean 2.x 就是"工具进步导致过度工程化"的反面教材。有了强大的 IDE 自动生成 XML 配置，人们觉得"配置多没关系"。结果 Spring 用"简单即正义"把 EJB 打得满地找牙。

### 6.5 人类能力的系统性退化风险

如果组织全面依赖 LLM + 复杂工程实践，新一代工程师可能失去设计简洁系统的能力。当 LLM 出现瓶颈时，组织发现自己既没有简单的代码，也没有能处理复杂代码的人。

## 七、最终立场：温和的激进主义

综合所有观点，我们的结论是：

### 应该做的（高确信度）

1. **用 LLM 降低"无侵入"实践的成本**：更好的测试、更好的文档、更好的 commit message、更强的 lint。这些不改变代码结构，退出成本为零。
2. **对最关键的 5% 代码使用形式化方法**：分布式共识协议、密码学实现、金融计算核心。其余 95% 用增强的测试和类型系统。
3. **Contract-First 架构**：人类写接口，LLM 写实现，测试验证合约。
4. **用类型系统而非命名约定**：如果要做效应系统/能力标记，让编译器强制执行，不要靠命名。
5. **增量生成，每步验证**：宁花 30 分钟做 6 个可信的小步骤，也不花 5 分钟做 1 个不可信的大步骤。

### 应该谨慎的（中确信度）

1. **契约式设计**：有价值，但需要工具验证，不能仅靠人工。
2. **属性测试**：有价值，但需要审查 LLM 生成的"性质"是否真正有意义。
3. **Result/Either 错误处理模式**：有价值，但注意不要让错误类型定义比业务代码还长。

### 应该避免的（高确信度）

1. **靠命名约定模拟效应系统**——没有编译器强制的约定必然腐化。
2. **一次性大规模生成**——审查成本比手写还高。
3. **自动生成 ADR / 决策记录**——编造历史比没有历史更危险。
4. **为 LLM 而拆微服务**——分布式复杂度远超 LLM 可靠处理能力。

### 一句话总结

> **LLM 是放大器——它放大好的实践的效果，也放大坏的做法的危害。不要因为有了一把更锋利的刀，就去做不必要的手术。**

## 附录：核心范式转换

| 维度 | 人类时代的妥协 | LLM 时代可以做到的 |
|------|---------------|-------------------|
| 代码量 | 精简优先 | 正确性优先，不怕膨胀 |
| 可读性 | 人类可读优先 | 审查可读性优先 |
| 抽象层 | 宁可低一些 | 可以适度提高 |
| 文档 | 能省则省 | 无侵入文档全自动生成 |
| 测试 | 只测 happy path | 穷举关键性质 |
| 类型 | 够用就行 | 关键路径用精化类型 |
| 契约 | 重要的加 | 关键函数都有前置/后置条件 |
| 错误处理 | 通用 Exception | 领域特定的穷举错误类型 |
| 重构 | 慎重 | 可以更频繁，但有验证保障 |
