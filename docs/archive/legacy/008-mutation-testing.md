# 变异测试 (Mutation Testing)

## 是什么

自动对源代码进行微小修改（"变异"），例如把 `>` 改成 `>=`、把 `+` 改成 `-`、把 `true` 改成 `false`，然后运行测试套件。如果测试仍然全部通过，说明测试没有覆盖到这个逻辑分支——这个"存活的变异体"暴露了测试的盲区。

变异测试衡量的不是代码覆盖率（哪些行被执行了），而是测试的**杀伤力**（测试真的在检查逻辑吗）。

## 历史相关渊源

概念最早由 Richard Lipton 在 1971 年提出，但因计算成本极高而停留在学术界。2000 年代中期，随着计算力提升和工具链成熟，实用工具开始涌现：Java 的 PIT (2010)、Ruby 的 Mutant (2012)、JavaScript 的 Stryker (2016)。

2008–2012 年间，变异测试在学术论文中被大量研究，Google 内部也在 2018 年发表了大规模变异测试的经验报告。但在开源社区和中小团队中，始终没有成为标准实践。

## TypeScript 代码举例

```typescript
// 原始代码
function isEligibleForDiscount(user: User): boolean {
  return user.age >= 65 || (user.isMember && user.purchaseCount > 10);
}

// ---- 变异测试工具自动生成的变异体 ----

// 变异体 1: >= 改为 > (边界条件变异)
function mutant1(user: User): boolean {
  return user.age > 65 || (user.isMember && user.purchaseCount > 10);
  //          ^^^ 65 岁的用户会被漏掉
}

// 变异体 2: || 改为 && (逻辑运算符变异)
function mutant2(user: User): boolean {
  return user.age >= 65 && (user.isMember && user.purchaseCount > 10);
  //                   ^^^ 必须同时满足年龄和会员条件
}

// 变异体 3: > 10 改为 >= 10 (边界条件变异)
function mutant3(user: User): boolean {
  return user.age >= 65 || (user.isMember && user.purchaseCount >= 10);
  //                                                           ^^^ 刚好 10 次也打折
}

// 变异体 4: 删除条件分支 (语句删除变异)
function mutant4(user: User): boolean {
  return true;
  //     ^^^^ 所有人都打折
}

// ---- 如果你的测试套件没有针对 age=65 的用例，变异体 1 就会存活 ----
// ---- 这说明你需要补一个边界测试 ----

// 杀死变异体 1 的测试：
// test("exactly 65 years old should be eligible", () => {
//   expect(isEligibleForDiscount({ age: 65, isMember: false, purchaseCount: 0 })).toBe(true);
// });
```

## 为什么被抛弃

1. **极其缓慢**：每个变异体都要跑一遍完整测试套件。一个中型项目可能生成数千个变异体，CI 时间从 5 分钟膨胀到数小时
2. **等价变异体问题**：有些变异实际上不改变程序行为（语义等价），无法被测试杀死，但工具无法自动区分，产生大量误报
3. **结果解读困难**："变异得分 73%"意味着什么？团队不知道该追求多少，也不知道哪些存活变异体值得修复
4. **投入产出比不明**：行覆盖率 90% 的团队再投入变异测试，边际收益递减
5. **工具生态不成熟**：Stryker (JS/TS) 的配置复杂，TypeScript 支持需要额外步骤，很多团队试了一次就放弃

## 是否应该在 LLM 时代重新启用

**应该，LLM 可以彻底改变变异测试的经济学。** 理由：

- **速度问题可以绕过**：LLM 不需要实际运行变异体。它可以通过静态分析判断"这个测试是否覆盖了这个逻辑分支"——不需要跑完整测试套件。
- **等价变异体问题**：LLM 能理解语义，可以自动识别并过滤等价变异体，大幅减少误报。
- **结果可自动消化**：LLM 可以把"存活变异体"直接转化为"应补充的测试用例"——不需要人去解读报告。
- **更本质的用途**：当 LLM 生成代码后，变异测试可以自动验证 LLM 同时生成的测试是否真的有意义——防止"全是 happy path 的废测试"。

建议：在 LLM 生成测试时，自动对被测代码做变异分析，确保生成的测试能杀死主要变异体。不需要在 CI 中全量跑，而是作为测试生成的质量保证。
