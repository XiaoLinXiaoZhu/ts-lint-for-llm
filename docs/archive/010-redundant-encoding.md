# 冗余信息编码（注释 / 命名 / 类型的重叠表达）

## 是什么

同一条语义信息通过多个通道同时表达：类型约束、命名前缀、注释说明三者传达相同的意思。目的是让信息在任何一个通道被忽略时，另一个通道仍然能传达。

这在传统编程中被视为"违反 DRY"的坏实践。但在 LLM 时代，多通道冗余可能是提高代码被正确理解概率的手段。

## 历史渊源

航空航天领域的"三重冗余"（Triple Modular Redundancy）是同一思想在硬件上的体现——三个模块做同样的计算，多数表决结果。

在软件中，经典的反面教材是：
```
int count = 0; // 计数器，初始化为 0
```
注释和代码说了完全一样的事，被视为噪音。

但在安全关键领域（MISRA C、DO-178B），要求代码中的约束同时在注释里用自然语言重述——因为审查者可能只看注释，也可能只看代码。

## 对人为什么太繁琐

- "这不就是把同一个东西写了三遍吗？"——维护时要同步更新三处
- DRY 原则深入人心，冗余 = 坏代码 = code review 会被指出
- 多数情况下类型已经足够精确，额外的注释和命名前缀是噪音
- 改了类型忘改注释 → 注释撒谎 → 比没有注释更糟

## LLM 为什么不怕

- LLM 可以自动保持三个通道的同步——改了类型就同步更新注释和命名
- 冗余信息对 LLM 是"多角度信号"——当 LLM 的上下文窗口不完整时（看不到类型定义），命名或注释中的冗余信息可以补偿
- LLM 生成代码时，如果目标函数的名字、类型、注释都在说"这个值已经校验过"，LLM 几乎不可能犯"再校验一次"的错误

## TypeScript 代码举例

```typescript
// ---- 单通道（只靠类型）----

function process(input: ValidatedEmail): UserId { /* ... */ }
// 如果 LLM 看不到 ValidatedEmail 的定义，它不知道这个值有什么属性

// ---- 三通道冗余 ----

/** 将已校验的邮箱地址关联到用户，返回用户 ID。邮箱必须已通过格式和域名校验。 */
function associateValidatedEmailToUser(validatedEmail: ValidatedEmail): UserId { /* ... */ }
// 通道 1（类型）: ValidatedEmail → 已校验
// 通道 2（命名）: associateValidatedEmail... → 名字里就说了"已校验的邮箱"
// 通道 3（注释）: "邮箱必须已通过格式和域名校验" → 自然语言解释约束

// 即使 LLM 的上下文中缺少 ValidatedEmail 的定义，
// 它仍然能从函数名和注释中理解"不需要再校验"

// ---- 在模块边界处更有价值 ----

/**
 * 系统入口：接收外部世界的原始 HTTP 请求体。
 * raw 参数完全不可信——可能包含任意数据、格式错误、恶意输入。
 * 返回 Result：成功则为已校验的注册信息，失败则为具体的校验错误。
 */
function parseRegistrationRequest(
  rawBody: unknown
): Result<ValidatedRegistration, RegistrationValidationError> {
  // ...
}
// 类型：unknown → 不可信
// 命名：rawBody → 原始的
// 注释：明确说"完全不可信"
// 三者共同传达同一个信息：这是系统边界，输入不可信

// ---- 对比：内部函数不需要冗余 ----

// 模块内部、参数已经是 branded type → 一个通道就够了
function calculateShipping(order: ValidatedOrder): ShippingCost {
  // 不需要注释说"order 已经校验过"——类型名已经说了
  // 不需要命名成 calculateShippingForValidatedOrder——过于冗长
  return /* ... */;
}

// ---- 规则：冗余程度与信任距离成正比 ----

// 系统边界（距外部世界最近）→ 三通道冗余
// 模块公开接口（可能被不熟悉内部的人调用）→ 类型 + 必要注释
// 模块内部私有函数（调用者和定义者同一个人/LLM）→ 类型名即可
```

## AI 回复

你贴的那段 SSOT 思辨非常有价值，让我把它拆解一下：

**核心链条**：代码是 SSOT → 知识在离它最近的代码处维护 → 模块应该自解释（不需要全局视角）→ 需要全局视角才能理解模块 = 耦合 → 全局协调通过依赖注入在 main 函数里做 → main 的注释就是全局文档 → PRD 最终应该体现为测试。

这条链条非常自洽，而且给出了一个精确的判定标准："如果你需要一份全局文档才能理解一个模块在干什么，说明这个模块和其他模块耦合了。"

关于"PRD 写进测试里"这个想法很有意思——测试用例本身就是"系统应该做什么"的可执行规格说明。PRD 说"用户输入无效邮箱时应该提示错误"，测试写 `expect(register({ email: "bad" })).toEqual({ ok: false, error: { kind: "invalid_email" } })`——两者说的是同一件事，但测试是可执行的、可验证的、不会过期的。

唯一的边界：AGENTS.md 作为"LLM 的行为配置"仍然是必要的例外——它不是描述代码做什么的文档，而是约束代码应该怎么写的规范。这属于"元信息"而非"内容信息"，可以独立于代码存在。
