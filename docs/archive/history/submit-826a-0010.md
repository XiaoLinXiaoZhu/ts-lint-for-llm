# ❓ 需要确认

## 问题

你指出了能力标注系统的根本局限：它只贴标签，不驱动改善。当前系统不能定义"好坏"、不能最小化副作用、不能告诉你"这个 IO 函数里有一半纯逻辑可以提出来"。你说"需要想想看"——你现在倾向于哪个方向？

## 选项

### 1. 探索能主动驱动副作用最小化的机制

> 比如分析 IO 函数内部结构，检测'被 IO 绑架的纯逻辑'并建议提取。需要更深的 AST 分析能力，和当前的标签式方案是不同维度的工具。

### 2. 先暂停工具开发，回到议题层面讨论'副作用隔离'这个代码写法应该怎么定义

> 回到 docs/ 的头脑风暴框架，把'副作用隔离最小化'作为一个新议题展开讨论，先想清楚目标再做工具。

### 3. 承认标注方案价值有限，换个完全不同的切入点

> 把能力标注作为'已探索但价值有限'的方案归档，重新思考什么工具/规范能真正驱动副作用隔离。

### 4. 你有别的想法，我听你说

> 你可能已经有了更具体的方向，我先听。
