# SSOT 规范：代码自文档化的边界与实践

> 来源：010（冗余编码）中"我的看法"的 SSOT 思辨链条。用户表示"需要保留，后续需要衍生讨论"。

## 核心思辨链条

原始讨论中提出的链条（从 010 迁移）：

1. **代码是 SSOT** → 尽量不要留文档在外面，README 最好也别有
2. **就近维护** → 所有东西在离它最近的代码处维护，改代码时顺手改注释，模型 grep 读文件时顺便读注释
3. **全局视角 = 耦合信号** → 如果要有全局视角才能搞懂模块有什么用，这不是耦合了吗
4. **依赖注入做全局协调** → main 函数里做依赖注入的那部分代码的注释，相当于全局文档
5. **PRD 写进测试** → 测试用例是可执行的、可验证的、不会过期的需求规格说明

## 这套原则在当前项目中的体现

- `prototype/src/capabilities.ts` 顶部注释 = 能力词汇表的 SSOT
- `prototype/scoring/engine.ts` 顶部注释 = 评分公式的 SSOT
- `.caps.ts` 声明文件 = 外部模块能力的 SSOT（类似 `@types/`）
- docs/ 目录 = 设计探索记录，不是代码文档（代码文档在代码里）

## 待衍生的讨论点

### 1. 对 LLM 的特殊意义

LLM 读代码的方式和人不同——它没有 IDE 的"跳转到定义"，依赖的是文本搜索和上下文窗口。SSOT 在代码中意味着 LLM 读到函数定义时就能看到所有关键信息（能力标注、数据阶段、错误类型），不需要跨文件跳转。

这和 012 评分模型的设计哲学一致：评分越低的代码 → 函数越小、能力越少 → 每个函数的上下文越自包含 → LLM 理解成本越低。

### 2. docs/ 的定位

如果代码是 SSOT，docs/ 应该放什么？

当前实践：docs/ 放的是**设计探索和决策记录**（类似 ADR），而不是代码的使用文档。使用文档在代码顶部注释中。这个边界是否合理？

### 3. main 函数作为全局文档

"main 函数的注释就是全局文档"这个想法在小项目里成立，但项目变大后 main 可能被拆成多个入口。全局协调的代码（依赖注入、模块组装）应该集中在什么地方？它的注释应该遵循什么规范？

### 4. 测试作为 PRD

"PRD 写进测试"意味着测试用例要有良好的命名和组织，让非技术人员也能从测试描述中理解系统行为。这对测试的写法有什么具体要求？

### 5. 与冗余编码（010）的张力

010 主张"类型、命名、注释三通道冗余表达"。但 SSOT 原则要求"信息只维护一处"。两者的边界在哪？

可能的区分：**同一个信息**不应该在多处维护（SSOT），但**同一个语义**可以在多个通道表达（冗余编码）。比如"这个参数是已验证的邮箱"可以同时通过类型（`ValidatedEmail`）、命名（`validatedEmail`）和 JSDoc（`@validated`）表达——三者说的是同一件事，但真正的维护只在类型定义处。
